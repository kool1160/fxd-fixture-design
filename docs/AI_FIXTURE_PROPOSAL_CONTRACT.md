# AI Fixture Proposal Contract

Milestone 31 introduces `fxd-fixture-proposal-v1`, a CAD-neutral, typed proposal
contract. It is review evidence, not fixture approval, released geometry, or a
replacement for deterministic engineering.

## Workflow and authority

1. The engineer accepts a source-SHA-linked manufacturing orientation.
2. FXD asks only for missing fixture purpose, process, handling mode, quantity,
   lifecycle, loading direction, and unloading direction. Recommended answers
   are visible and require an explicit apply action.
3. Existing placement, concept, access, weld, and validation engines prepare the
   governed context.
4. A configured provider may return one strict JSON proposal. Unknown fields,
   types, identities, source hashes, orientation identities, or engineering-
   context identities quarantine the response.
5. Deterministic validation evaluates every accepted proposal. Its blockers and
   warnings override AI confidence and ranking.
6. The engineer may accept, reject, suppress, or edit recommendations. Each
   decision creates audit evidence and invalidates downstream authored fixture
   geometry, drawing intent, cost intent, approval, and export eligibility.

The proposal covers orientation, datum strategy, locators, supports, clamps,
base structure, weld or tack access, load and unload, operator or automation
access, standard tooling, lifecycle, unresolved questions, and alternatives.
Each recommendation records source evidence, assumptions, confidence,
deterministic checks, unresolved risk, editable parameters, downstream
dependencies, and an optional stable geometry or fixture-feature identity.

## Provider boundary

`AiFixtureProvider` is provider-neutral. The shipped HTTP JSON adapter reads
configuration only from the process environment:

- `FXD_AI_ENDPOINT`
- `FXD_AI_API_KEY`
- `FXD_AI_MODEL`
- optional `FXD_AI_PROVIDER`

Credentials are not written to projects, diagnostics, prompts, or repository
files. Requests contain compact structured identities, bounds, intent,
orientation axes, annotations, sanitized customer-tooling evidence, placements,
deterministic candidates, findings, and alternatives. Tooling source paths, STEP
bytes, base64 source geometry, unrestricted project dumps, customer credentials,
and hidden chain-of-thought are excluded. Unverified customer-tooling records are
excluded from the provider payload and strict identity set; only verified records
may be cited by provider recommendations.

Provider calls are bounded by a caller-visible timeout and cooperative
cancellation. Missing configuration produces the explicit state `ai_unavailable`.
Timeout, provider failure, or quarantined output produces
`proposal_generation_failed`. When fallback is allowed, both states return an
accurately labeled deterministic baseline generated by existing FXD engines;
they never claim that AI generated it. Tests use `StaticAiProvider`, not a paid
or network service.

## Persistence and stale state

Projects use `fxd-neutral-project-v5` and retain v1 through v4 read
compatibility. The persisted proposal includes schema and proposal identities,
source SHA-256, manufacturing-orientation identity, governed engineering-context
identity, provider and model provenance, prompt and response contract versions,
timestamp, recommendations, guided issues, engineer decisions and edits,
overall decision, and regeneration history.

A source mismatch fails attachment or load closed. An orientation, manufacturing-
intent, annotation, customer-tooling, placement, concept, or upstream deterministic-evidence
mismatch keeps the old proposal visible as stale. Visible intent controls are
persisted before an asynchronous request is bound, and a completion is discarded
if that governed context or authored project identity changes while the request
runs. Stale proposals and
proposals with deterministic blockers cannot be accepted, approved, or exported.
Source STEP bytes and source coordinates remain unchanged.

Generating or regenerating a proposal always re-runs deterministic analysis for
the current workflow. For an existing project, FXD then deterministically replays
and verifies its active concept, edit log, suppressed features, review decisions,
layers, revision history, and immutable product evidence. Proposal replacement
still invalidates downstream authored fixture, drawing, optimization, approval,
and export evidence by design.

## Guided validation

Every guided issue contains severity, a plain-language title, what is wrong,
why it matters, rule ID, affected identity when available, workflow section,
correction target, and optional technical evidence. The desktop `Fix this`
action navigates to the responsible workflow section, focuses the mapped input,
and highlights linked model evidence. Re-running proposal validation selects the
next unresolved issue. Technical rule evidence remains behind `More details`.

## Limits

Milestone 31 does not add conversational chat, autonomous iteration, learned
global rules, supplier scraping, paid-provider provisioning, cloud project
storage, production release, structural certification, weld-process
qualification, safety certification, or robot-path simulation. Provider
cancellation cannot interrupt an operating-system HTTP call already in flight;
the configured timeout bounds it and the result is discarded after cooperative
cancellation.
